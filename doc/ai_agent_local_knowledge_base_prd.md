# AI Agent 本地知识库工具 PRD

## 1. 产品概述

### 1.1 产品名称
暂定名：**KnowPilot AI 本地知识库**

### 1.2 产品定位
KnowPilot 是一个面向个人和小团队的 **AI Agent 驱动型本地知识库工具**。它不仅是一个文档管理系统，也不是简单的 RAG 问答工具，而是一个能够完成“资料采集 → 文档解析 → 知识结构化 → 实体关系抽取 → 图谱构建 → 深度研究 → 知识再沉淀 → 日常检索与创作辅助”的智能知识工作台。

核心目标：

> 帮用户把散落在网页、PDF、Word、Excel、图片、搜索结果、研究报告、聊天记录中的知识，自动变成可搜索、可追踪、可编辑、可推理、可复用的个人知识资产。

### 1.3 产品愿景
打造一个“本地优先、AI 增强、图谱驱动、持续生长”的个人知识系统，让用户在日常学习、研究、投资、技术排查、项目管理和资料积累中，不再只是保存文件，而是沉淀可理解、可关联、可追问、可复用的知识网络。

### 1.4 目标用户

| 用户类型 | 典型需求 |
|---|---|
| 技术研发人员 | 保存技术文档、源码笔记、问题排查记录、架构资料，支持快速定位和关联分析 |
| 投资/研究人员 | 收集行业报告、财报、新闻、公司资料，自动形成公司/人物/事件/产业链图谱 |
| 学习型用户 | 管理课程、论文、书籍、网页，生成摘要、卡片、复习题、知识地图 |
| 产品经理/运营人员 | 收集竞品、需求、访谈、市场资料，辅助输出 PRD、调研报告、方案 |
| 小团队 | 共享知识库、统一标签分类、支持权限和协作备注 |

## 2. 用户痛点

### 2.1 资料多，但无法真正沉淀
用户经常保存网页、PDF、Word、Excel、图片、截图、报告，但后续很难找到，也很难知道这些资料之间的关系。

### 2.2 普通知识库只存文档，不理解内容
传统知识库一般只做全文搜索和标签管理，无法自动识别实体、关系、事件、观点、结论、数据来源。

### 2.3 RAG 问答容易“只回答，不沉淀”
很多 AI 知识库支持问答，但问答结果没有被结构化保存，用户下一次仍要重新问，缺少持续积累机制。

### 2.4 图谱工具和文档工具割裂
图数据库能表达实体关系，但普通用户很难手工维护；文档系统又无法自然展示实体网络。

### 2.5 网络搜索和本地资料割裂
用户经常需要基于本地知识继续查网络资料，但搜索结果、网页摘要、引用来源没有自动沉淀回本地知识库。

### 2.6 NotebookLM 很好用，但本地知识资产不可控
NotebookLM 适合基于资料做学习和问答，但用户仍需要一个自己的本地知识底座，管理原始文档、结构化结果、图谱、版本和长期沉淀。

## 3. 产品核心原则

1. **本地优先**：文档、向量、图谱、索引优先存储在本地，用户可控。
2. **来源可追溯**：每个 AI 结论、实体、关系、摘要必须能回溯到原文片段。
3. **先存证，再推理**：先保存原文和解析结果，再进行 Agent 分析，避免“只生成不留痕”。
4. **自动化但可编辑**：分类、摘要、实体、关系、标签都可自动生成，但用户必须能修改。
5. **多视图统一**：同一份知识可在文档视图、卡片视图、图谱视图、时间线视图、问答视图中查看。
6. **知识持续生长**：深度研究、网络搜索、问答结果、备注都可以再次导入知识库。
7. **可插拔模型**：支持本地模型和云端模型混合，低成本使用，隐私内容可走本地。

## 4. 产品范围

### 4.1 MVP 版本范围
MVP 先解决“个人本地知识库 + AI 解析 + 图谱 + 问答 + 标注”的核心闭环。

MVP 必须包含：

1. 多格式导入：URL、Word、Excel、PDF、图片、Markdown、TXT。
2. 文档解析：文本提取、OCR、表格提取、章节识别。
3. 文档编辑：解析后可在线修改并保存。
4. 自动分类：最多 3 级分类。
5. 实体识别：人物、组织、地点、产品、技术、事件、时间、指标等。
6. 关系抽取：实体之间的关系抽取并写入图数据库。
7. 图谱展示：节点、边、详情面板、节点扩展。
8. 划线标记和备注：支持高亮、下划线、评论、摘录卡片。
9. 本地知识库问答：基于文档和引用来源回答。
10. 网络搜索：文档级搜索、实体级搜索、深度研究搜索。
11. 深度研究：自动拆解问题、搜索、阅读、归纳、输出报告。
12. 研究结果再导入：报告、引用来源、实体关系回流知识库。

### 4.2 非 MVP 但重要版本

1. NotebookLM Enterprise API 对接。
2. 浏览器插件。
3. 移动端采集。
4. 多人协作和权限。
5. 时间线知识图谱。
6. 事件抽取与因果链分析。
7. 自动生成播客/学习卡/测验题。
8. 本地模型管理中心。
9. 第三方知识库迁移：Obsidian、Notion、Zotero、Logseq、OneNote。

## 5. 核心用户场景

### 5.1 场景一：导入行业报告并自动生成知识图谱
用户上传 PDF 行业报告，系统自动识别公司、产品、政策、市场规模、产业链上下游关系，生成图谱。用户点击某个公司节点，可以查看原文出处、相关文档、网络解释和深度研究入口。

### 5.2 场景二：网页资料剪藏并持续沉淀
用户输入 URL 或通过浏览器插件保存网页。系统抓取正文、清洗广告、生成摘要、识别实体、归类到“AI/大模型/Agent”分类下，并关联已有知识。

### 5.3 场景三：技术问题排查知识库
用户导入日志、技术文档、Shell 脚本、错误截图。系统自动识别错误类型、服务组件、IP、端口、配置项、解决步骤。后续用户提问“这个错误之前怎么解决的”，系统可以直接返回历史方案。

### 5.4 场景四：深度研究并回填知识库
用户输入“AI Agent 本地知识库有哪些竞品和技术路线”。Agent 自动制定研究计划，搜索资料，读取网页，提取竞品、功能、技术栈、差异点，生成报告。用户点击“导入知识库”，系统将报告、引用来源、实体关系、竞品卡片全部入库。

### 5.5 场景五：读文档时做标注和笔记
用户阅读 PDF 或网页正文时，对关键段落划线、添加备注。备注可以被 AI 总结成知识卡片，也可以加入专题集合。

### 5.6 场景六：结合 NotebookLM 使用
用户在本地知识库中整理一批资料，选择一个专题，一键导出为 NotebookLM 适配格式。企业版用户可通过 API 创建 notebook 并添加数据源；个人版用户可导出 Google Docs、PDF、Markdown 或复制文本手动导入。

## 7. 详细功能设计

## 7.1 知识导入中心

### 7.1.1 支持导入格式

| 类型 | 格式 | 解析能力 |
|---|---|---|
| 文档 | doc / docx / odt | 提取标题、段落、表格、图片、批注，转换为结构化 Markdown |
| 表格 | xls / xlsx / csv | 提取 sheet、表头、数据区域、字段说明，支持表格问答 |
| PDF | 普通 PDF / 扫描 PDF | 文本提取、OCR、版面识别、页码引用、表格提取 |
| 图片 | png / jpg / jpeg / webp / tiff | OCR、图像描述、图中实体识别、截图内容理解 |
| 网页 | URL / HTML | 正文提取、标题、发布时间、作者、链接、图片、网页快照 |
| 文本 | txt / md / rst | 保留 Markdown 结构，自动分块 |
| 代码 | py / java / sh / yaml / json / sql | 代码块识别、函数/类/配置项抽取 |
| 音视频扩展 | mp3 / mp4 / YouTube URL | 语音转写、摘要、章节切分，作为后续版本功能 |

### 7.1.2 导入方式

1. 拖拽上传。
2. 点击上传。
3. URL 粘贴导入。
4. 文件夹批量导入。
5. 监听本地目录自动导入。
6. 浏览器插件一键剪藏。
7. 粘贴文本快速创建文档。
8. 深度研究结果一键导入。

### 7.1.3 导入任务状态

任务状态：

```text
等待中 → 下载/读取中 → 解析中 → AI 理解中 → 入库中 → 完成
                                ↓
                              失败 / 可重试
```

任务详情展示：

- 文件名 / URL
- 文件大小
- 导入时间
- 解析耗时
- 提取字符数
- 识别实体数
- 识别关系数
- 分类结果
- 错误日志

### 7.1.4 导入后自动处理流程

```text
原始文件
  ↓
文件指纹计算，避免重复导入
  ↓
文本 / 表格 / 图片 / OCR 提取
  ↓
结构化 Markdown 保存
  ↓
语义分块
  ↓
摘要、关键词、分类、标签生成
  ↓
实体识别和关系抽取
  ↓
写入全文索引、向量库、图数据库
  ↓
生成文档详情页和图谱入口
```

## 7.2 文档解析与编辑

### 7.2.1 原始文档与解析文档分离
每个导入文档保存两类内容：

1. **原始文件**：不可修改，用于追溯。
2. **解析文档**：可编辑的结构化内容，一般使用 Markdown + 元数据保存。

这样可以避免用户修改后丢失原始证据。

### 7.2.2 在线编辑能力

支持：

- 修改标题。
- 修改正文。
- 修改表格。
- 添加、删除、合并段落。
- 手动调整分类。
- 添加标签。
- 修改 AI 摘要。
- 手动修正实体识别结果。
- 手动修正关系。
- 保存为新版本。

### 7.2.3 版本管理

每次保存生成版本：

| 字段 | 说明 |
|---|---|
| version_id | 版本 ID |
| doc_id | 文档 ID |
| version_no | 版本号 |
| content_hash | 内容哈希 |
| change_summary | 修改摘要 |
| created_at | 保存时间 |
| created_by | 修改人 |

支持：

- 查看历史版本。
- 对比版本差异。
- 回滚版本。
- 重新执行 AI 解析。

## 7.3 自动分类体系

### 7.3.1 分类层级
分类最多 3 级：

```text
一级分类 / 二级分类 / 三级分类
```

示例：

```text
技术 / 大数据 / Hadoop
技术 / AI / Agent
投资 / 公司研究 / 英伟达
投资 / 行业研究 / 半导体
生活 / 学习 / 英语
项目 / WorldMonitor / MinIO
```

### 7.3.2 自动分类策略

AI 自动分类时参考：

1. 文档标题。
2. 正文摘要。
3. 关键词。
4. 实体类型。
5. 用户已有分类树。
6. 历史相似文档。
7. 用户手动修正记录。

### 7.3.3 分类推荐结果

系统给出：

```text
推荐分类：技术 / AI / Agent
置信度：86%
理由：文档多次出现 Agent、RAG、工具调用、任务规划、知识库等关键词。
可选分类：
1. 技术 / AI / RAG
2. 产品 / 知识管理 / AI 工具
3. 技术 / 图数据库 / 知识图谱
```

### 7.3.4 用户反馈学习
如果用户多次把“MinIO、Docker、FastAPI”相关文档归到“项目 / WorldMonitor / 存储服务”，系统后续应优先推荐该分类。

## 7.4 实体识别与关系抽取

### 7.4.1 实体类型

实体识别不应设计成固定死的枚举，而应采用 **基础实体类型 + 领域实体类型 + 用户自定义实体类型 + AI 自动发现实体类型** 的体系。

#### 7.4.1.1 基础实体类型

|---|---|
| 人物 | Sam Altman、张三、黄仁勋 |
| 公司/组织 | OpenAI、Google、阿里云、英伟达 |
| 产品 | NotebookLM、ChatGPT、MinIO、H100 |
| 技术 | RAG、GraphRAG、OCR、FastAPI、CUDA |
| 项目 | WorldMonitor、Hadoop 集群迁移 |
| 地点 | 北京、硅谷、土耳其 |
| 时间 | 2026 年 5 月、Q1、昨天 |

| 配置项 | fs.defaultFS、MINIO_ENDPOINT |
| 文件/路径 | /data/minio、docker-compose.yml |
| 错误信息 | ConnectTimeoutError、Mapper conflict |

| 商品/原材料 | 原油、铜、锂、硅料、DRAM | 产业链成本或价格变量 |
| 市场数据 | 股价、市值、成交量、换手率、涨跌幅 | 股票行情类数据 |
| 研究观点 | 买入评级、增持评级、目标价、风险提示 | 研报和分析文章中的观点 |
| 催化因素 | 新品发布、订单增长、政策落地、产能释放 | 影响股票或行业的潜在事件 |
| 风险因素 | 需求下滑、价格战、制裁、技术替代 | 投资和产业分析中的风险项 |

#### 7.4.1.3 用户自定义实体类型

用户可以根据自己的知识库创建实体类型，例如：


每个自定义实体类型应支持：

- 类型名称。
- 类型描述。
- 示例实体。
- 同义词。
- 识别规则。

#### 7.4.1.4 AI 自动发现实体类型

系统需要支持“实体识别类型自动丰富”，即当知识库中反复出现某类新概念时，AI 能自动建议创建新的实体类型。

示例：

```text
系统在多篇文档中发现：
- fs.defaultFS
- dfs.nameservices
- dfs.ha.namenodes
- yarn.resourcemanager.address

AI 建议新增实体类型：Hadoop 配置项
理由：这些实体都属于 Hadoop 配置参数，经常出现在集群部署和故障排查文档中。
```

再如：

```text
系统在多篇投资资料中发现：
- 毛利率
- ROE
- 自由现金流
- 资产负债率
- 存货周转率

AI 建议新增实体类型：财务指标
理由：这些实体都是用于公司基本面分析的指标。
```

自动发现流程：

```text
文档导入
  ↓
基础 NER 识别
  ↓
未归类名词短语抽取
  ↓
高频候选实体聚类
  ↓
LLM 判断候选实体共同上位概念
  ↓
生成“新实体类型建议”
  ↓
用户确认 / 修改 / 拒绝
  ↓
写入实体类型库
  ↓
后续识别自动使用新类型
```

#### 7.4.1.5 实体类型自动丰富算法设计

推荐采用多策略融合，而不是只依赖 LLM。

1. **规则识别**
   - 股票代码正则：`AAPL`、`NVDA`、`600519.SH`、`00700.HK`。
   - IP/端口正则。
   - 配置项正则。
   - 文件路径正则。
   - 财务指标词典。
   - 产业链环节词典。

2. **词典识别**
   - 公司简称词典。
   - 股票代码词典。
   - 行业分类词典。
   - 证监会行业分类。
   - GICS 行业分类。
   - 申万行业分类。
   - 技术名词词典。

3. **统计抽取**
   - 高频名词短语。
   - TF-IDF 关键词。
   - TextRank 关键词。
   - 共现关系。
   - 文档集合中的主题聚类。

4. **Embedding 聚类**
   - 对未归类实体生成向量。
   - 聚类相似实体。
   - 判断是否形成稳定类型。
   - 自动生成类型名称和解释。

5. **LLM 归纳**
   - 给定一组候选实体，让 LLM 判断它们是否属于同一类。
   - 让 LLM 给出类型名称、定义、示例、识别规则。
   - 要求 LLM 给出证据和置信度。

6. **用户反馈学习**
   - 用户确认的新类型进入实体类型库。
   - 用户拒绝的类型降低权重。
   - 用户手动修改的类型成为后续 few-shot 示例。

## 7.5 图数据库入库设计

### 7.5.1 推荐图数据库
个人本地优先推荐：

1. **Neo4j Community**：生态成熟、可视化友好。
2. **NebulaGraph**：适合大规模图数据，用户已有使用经验。
3. **KuzuDB**：嵌入式图数据库，适合本地应用。
4. **SQLite + graph extension / 自定义边表**：MVP 快速实现可选。

考虑用户已有 Nebula Graph 使用经验，产品可优先支持 NebulaGraph，同时保留 Neo4j 适配层。

### 7.5.2 图谱数据模型

节点：

```text
Entity
Document
Chunk
Annotation
ResearchReport
WebPage
Category
Tag
```

边：

```text
MENTIONED_IN：实体出现在文档中
HAS_CHUNK：文档包含分块
RELATED_TO：实体相关
BELONGS_TO：文档属于分类
HAS_TAG：文档拥有标签
ANNOTATED：用户标注文档片段
DERIVED_FROM：研究报告来源于网页/文档
SUPPORTS：证据支持结论
CONFLICTS_WITH：观点冲突
```

### 7.5.3 图谱写入策略

1. 原文导入后先生成 Document 和 Chunk。
2. NER 结果生成 Entity。
3. RE 结果生成 Entity-Entity 边。
4. 每个关系边必须关联 evidence chunk。
5. 用户修改实体或关系后，标记为 user_verified。
6. 低置信度关系进入待审核队列。

## 7.11 NotebookLM 对接

### 7.11.1 对接原则
NotebookLM 不作为本产品的数据主库，而作为外部学习和资料理解工具。KnowPilot 负责本地资产沉淀、结构化、图谱和版本；NotebookLM 负责对选定资料包进行外部学习、问答和生成。

### 7.11.2 个人版兼容方案
面向普通用户：

1. 从本地知识库选择一个专题。
2. 导出为 NotebookLM 友好格式：
   - PDF
   - Google Docs
   - Markdown
   - 复制文本
   - URL 列表
3. 用户手动上传到 NotebookLM。
4. 系统保存导出记录和资料包版本。
5. 用户可把 NotebookLM 生成的内容复制回 KnowPilot。
6. KnowPilot 对回填内容重新解析入库。

### 7.11.3 企业版 API 方案
面向企业版用户：

1. 配置 Google Cloud / Gemini Enterprise 凭据。
2. 创建 Notebook。
3. 添加数据源。
4. 维护本地专题与 Notebook 的映射。
5. 保存同步状态。
6. 支持重新导出或增量更新。

### 7.11.4 数据包设计

```text
NotebookLM 资料包
├─ README.md：资料说明
├─ sources/
│  ├─ doc_001.pdf
│  ├─ doc_002.md
│  └─ web_003.md
├─ index.md：资料目录
├─ entities.md：关键实体表
├─ timeline.md：关键事件时间线
└─ questions.md：建议提问列表
```

### 7.11.5 回链能力
每个导出资料包在本地记录：

- 导出时间。
- 导出文档列表。
- 导出版本。
- Notebook 名称。
- Notebook 链接。
- 回填记录。

## 13. 模型能力配置

### 13.1 模型类型

| 能力 | 模型类型 |
|---|---|
| 问答/总结 | LLM |
| 实体识别 | LLM + NER 模型 |
| 关系抽取 | LLM + 规则校验 |
| 向量检索 | Embedding 模型 |
| OCR | OCR 模型 |
| 图像理解 | 多模态模型 |
| 重排序 | Reranker 模型 |

### 13.2 推荐模型策略

个人本地：

- LLM：Qwen、Llama、Gemma 等本地可运行模型。
- Embedding：bge、gte、m3e、Qwen embedding。
- Reranker：bge-reranker。
- OCR：PaddleOCR。

### 14.1 导入文档流程

```text
用户上传文件
  ↓
系统识别格式
  ↓

  ↓
开始解析
  ↓
展示进度
  ↓

  ↓

  ↓
```

```text
用户点击实体
  ↓

  ↓

  ↓
系统搜索权威来源
  ↓

  ↓
用户选择是否保存
  ↓
保存为实体补充说明或新文档
```

### 14.3 深度研究回流流程

```text
用户创建研究问题
  ↓
Agent 生成研究计划
  ↓
本地搜索 + 网络搜索
  ↓
阅读来源并提取证据
  ↓
生成研究报告
  ↓
用户点击导入知识库
  ↓
系统创建报告文档
  ↓
抽取实体和关系
  ↓
更新图谱
```

## 18. 产品差异化

与普通知识库相比：

- 不只是存文档，而是自动抽取实体关系。
- 不只是 RAG，而是 RAG + 图谱 + 标注 + 深度研究。
- 不只是搜索，而是把搜索结果沉淀回知识库。
- 不只是 AI 问答，而是每个回答都可追溯、可保存、可再利用。
- 不只是本地工具，而是能与 NotebookLM 等外部工具形成协同。

与 NotebookLM 相比：

- 本地数据资产更可控。
- 支持图数据库和实体关系长期沉淀。
- 支持文档修改、版本管理、标注和知识卡片。
- 支持深度研究结果结构化入库。
- 支持自定义模型和私有部署。

与 Obsidian/Notion 相比：

- 自动解析和知识结构化更强。
- 图谱不是手工链接，而是实体关系驱动。
- 支持本地 AI Agent 工作流。
- 支持多格式自动导入和 OCR。

## 19. 风险与解决方案

### 19.1 关系抽取准确率风险
风险：LLM 抽取关系容易出现错误。

解决：

- 所有关系绑定原文证据。
- 设置置信度。
- 低置信度进入待审核。
- 用户可编辑关系。
- 使用规则和 schema 校验。

### 19.2 图谱过度复杂
风险：节点太多，图谱不可读。

解决：

- 默认只显示核心节点。
- 支持按类型和关系筛选。
- 支持一跳展开。
- 支持社区聚类。
- 支持图谱摘要。

### 19.3 多格式解析难度高
风险：PDF、表格、扫描件解析效果不稳定。

解决：

- MVP 优先支持常见格式。
- 解析失败允许手动修正。
- 保存原始文件。
- 解析管线可替换。

### 19.4 NotebookLM API 不确定性
风险：个人版 API 能力有限或不可用。

解决：

- 个人版采用导出资料包方案。
- 企业版再接官方 API。
- 本产品不依赖 NotebookLM 作为核心功能。

### 19.5 本地模型能力不足
风险：本地模型在深度研究、关系抽取方面效果不如云模型。

解决：

- 支持本地 + 云混合。
- 敏感数据本地处理。
- 普通任务走低成本模型。
- 复杂研究可选择高能力云模型。
