# 工程开发规范

> 项目：KnowPilot AI Agent 本地知识库工具  
> 用途：指导产品、前端、后端、算法、测试和运维团队协同研发  
> 版本：v0.1  
> 日期：2026-05-15


## 1. 总体原则

1. Schema 优先，接口和数据结构先定义再实现。
2. 后端业务逻辑不得直接散落在 API 层。
3. 模型调用必须通过统一抽象接口。
4. 数据库变更必须走迁移。
5. 任何 AI 输出都必须经过校验。
6. 日志必须能定位任务和请求。

---

## 2. Git 分支规范

| 分支 | 用途 |
|---|---|
| main | 稳定发布分支 |
| develop | 日常集成分支 |
| feature/* | 功能开发 |
| bugfix/* | 缺陷修复 |
| hotfix/* | 紧急修复 |
| release/* | 发布准备 |

提交格式：

```text
feat(document): add file import api
fix(parser): handle empty pdf text
refactor(search): split hybrid search service
docs(api): update document import spec
test(entity): add stock entity extraction tests
```

---

## 3. 后端目录规范

```text
backend/
  app/
    api/v1/
    core/
    domain/
    services/
    infrastructure/
    workers/
    schemas/
    tests/
```

规则：

1. `api` 只负责参数接收和响应。
2. `services` 负责业务编排。
3. `domain` 放领域模型和领域规则。
4. `infrastructure` 放数据库、向量库、图数据库、模型客户端。
5. `schemas` 放 Pydantic 请求响应模型。
6. `workers` 放异步任务。

---

## 4. 前端目录规范

```text
frontend/
  src/
    pages/
    components/
    services/
    stores/
    types/
    hooks/
    utils/
```

规则：

1. 页面级组件放 pages。
2. 可复用组件放 components。
3. 接口调用放 services。
4. 全局状态放 stores。
5. TypeScript 类型放 types。
6. 页面不得直接写复杂请求逻辑。

---

## 5. API 命名规范

1. 路径使用复数名词。
2. 版本统一 `/api/v1`。
3. 查询使用 GET。
4. 创建使用 POST。
5. 修改使用 PUT/PATCH。
6. 删除使用 DELETE。
7. 长任务创建接口必须返回 job_id。

示例：

```http
POST /api/v1/documents/import/file
GET /api/v1/documents/{doc_id}
PUT /api/v1/documents/{doc_id}/content
POST /api/v1/entities/merge
```

---

## 6. 数据库规范

1. 所有表使用 snake_case。
2. 主键使用字符串 ID。
3. 时间字段统一 `TIMESTAMPTZ`。
4. JSON 扩展字段使用 `JSONB`。
5. 重要表必须有 `created_at` 和 `updated_at`。
6. 业务删除优先软删除。
7. 所有迁移必须可重复执行和回滚。

---

## 7. 异常处理规范

后端异常统一转成错误码：

```json
{
  "success": false,
  "error": {
    "code": "PARSE_FAILED",
    "message": "PDF 解析失败",
    "details": {
      "doc_id": "doc_001"
    }
  }
}
```

不得直接把 Python 堆栈返回给前端。

---

## 8. 日志规范

日志必须包含：

- trace_id
- workspace_id
- user_id
- job_id
- doc_id
- task_type
- model_name

日志级别：

| 级别 | 用途 |
|---|---|
| DEBUG | 开发调试 |
| INFO | 正常业务流程 |
| WARNING | 可恢复异常 |
| ERROR | 任务失败 |
| CRITICAL | 系统不可用 |

---

## 9. 模型调用规范

1. 业务代码不得直接调用具体模型厂商 SDK。
2. 必须通过 LLMClient、EmbeddingClient、RerankerClient、OCRClient。
3. 所有结构化输出必须 Schema 校验。
4. 模型调用必须记录耗时、token、模型名称。
5. 敏感文档必须检查 sensitive_level。

---

## 10. Prompt 规范

1. Prompt 必须有名称和版本。
2. Prompt 变更必须记录 changelog。
3. Prompt 输出必须有测试样本。
4. 不允许在代码中硬编码大段 Prompt。
5. Prompt 中必须明确“不确定则不要输出”。

---

## 11. 前端交互规范

1. 所有按钮点击后要有反馈。
2. 所有长任务要有进度。
3. 所有删除要二次确认。
4. 所有 AI 结果要支持重新生成。
5. 所有列表要支持分页或虚拟滚动。
6. 表单要做必填校验。

---

## 12. 测试规范

1. 后端服务函数必须写单元测试。
2. 核心 API 必须写接口测试。
3. AI Pipeline 必须准备样本测试。
4. 前端关键页面必须有基本渲染测试。
5. 发布前必须跑回归测试。

---

## 13. 环境变量规范

示例：

```env
APP_ENV=development
DATABASE_URL=postgresql://user:pass@localhost:5432/knowpilot
REDIS_URL=redis://localhost:6379/0
QDRANT_URL=http://localhost:6333
MINIO_ENDPOINT=localhost:9000
MINIO_ACCESS_KEY=knowpilot
MINIO_SECRET_KEY=knowpilot_secret
OLLAMA_BASE_URL=http://localhost:11434
DEFAULT_LLM_MODEL=qwen
DEFAULT_EMBEDDING_MODEL=bge-m3
```

敏感信息不得提交到 Git。

---

## 14. Docker 规范

1. 每个服务独立 Dockerfile。
2. docker-compose 用于本地开发。
3. 生产环境配置单独维护。
4. 数据目录必须挂载到宿主机。
5. 日志输出到 stdout/stderr。

---

## 15. 代码评审清单

- 是否符合目录规范？
- 是否有测试？
- 是否处理异常？
- 是否有日志？
- 是否影响数据库迁移？
- 是否影响 Prompt 或 Schema？
- 是否有性能风险？
- 是否会泄露敏感信息？
